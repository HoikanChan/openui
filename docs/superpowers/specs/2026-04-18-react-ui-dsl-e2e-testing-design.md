# E2E Testing Design for `@openuidev/react-ui-dsl`

**Date:** 2026-04-18  
**Scope:** Table + Chart components (PieChart, LineChart, BarChart, GaugeChart)  
**Goal:** Catch both parse errors and render crashes using LLM-generated DSL snapshots

---

## Problem

The `react-ui-dsl` package has two categories of errors:

1. **Parse errors** — DSL strings generated by LLMs fail schema validation in `createParser`
2. **Render errors** — React components crash or produce wrong output during rendering

The existing Table unit test (`Table/index.test.tsx`) tests `mapColumnsToAntd` logic in isolation. There is no test that validates the full `openui-lang DSL → parse → render` pipeline with realistic LLM-generated input.

---

## Approach: VCR-style LLM snapshot testing

Tests are driven by real LLM-generated DSL output, stored as snapshot files. Each test run reads the snapshots; the LLM is only called when a snapshot is missing or regeneration is explicitly requested.

```
First run (no snapshot):
  prompt + dataModel → Claude API → DSL string → saved to .dsl file → parse + render → assert

Subsequent runs:
  .dsl file → parse + render → assert   (no LLM call)

Regeneration:
  REGEN_SNAPSHOTS=1 pnpm test:e2e → calls LLM for every fixture, overwrites .dsl files
```

This approach:
- Tests realistic LLM output, not hand-crafted DSL
- Is deterministic after the first run (no flakiness from LLM)
- Makes snapshot diffs reviewable in git (`*.dsl` files are plain text)

---

## Technology Stack

| Layer | Tool | Reason |
|---|---|---|
| Test runner | **vitest** (existing) | `describe.each` / `it.each` for parameterized tests |
| Component rendering | **@testing-library/react** + jsdom | Runs `useEffect`, real DOM nodes — community standard |
| LLM calls | **openai** SDK | Consistent with project examples (`examples/*/api/chat/route.ts`) |
| ECharts isolation | `vi.mock('echarts')` | `echarts.init` requires canvas; mock verifies props are passed, not canvas drawing |
| Snapshot storage | Plain `.dsl` text files | Human-readable, git-diffable |

## Fixtures

Each fixture has:
- `id` — snapshot filename stem
- `prompt` — plain-language description sent to the LLM
- `dataModel` — realistic API response shape (not component-specific); the LLM must map `data.xxx` paths to component props
- `assert.contains` — strings the rendered DOM must contain

## LLM Helper (`llm.ts`)

The system prompt includes:
1. `dslLibrary.toSpec()` — component signatures the LLM must follow
2. `dataModel` structure — available `data.xxx` paths
3. Instruction to return only DSL, no explanation

```ts
import { existsSync, mkdirSync, readFileSync, writeFileSync } from "node:fs";
import { resolve } from "node:path";
import OpenAI from "openai";
import type { PromptSpec } from "@openuidev/lang-core";

const SNAPSHOT_DIR = resolve(__dirname, "snapshots");

export async function loadOrGenerate(
  id: string,
  prompt: string,
  dataModel: Record<string, unknown>,
  spec: PromptSpec,
): Promise<string> {
  const snapshotPath = resolve(SNAPSHOT_DIR, `${id}.dsl`);

  if (!process.env.REGEN_SNAPSHOTS && existsSync(snapshotPath)) {
    return readFileSync(snapshotPath, "utf-8");
  }

  const dsl = await callLLM(prompt, dataModel, spec);
  mkdirSync(SNAPSHOT_DIR, { recursive: true });
  writeFileSync(snapshotPath, dsl, "utf-8");
  return dsl;
}

async function callLLM(
  prompt: string,
  dataModel: Record<string, unknown>,
  spec: PromptSpec,
): Promise<string> {
  const client = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });

  const systemPrompt = `You generate openui-lang DSL.

Available components:
${JSON.stringify(spec, null, 2)}

The user has runtime data accessible via data.xxx with this shape:
${JSON.stringify(dataModel, null, 2)}

Rules:
- Reference data using data.fieldName paths (e.g. data.report.breakdown)
- Return ONLY the DSL. No explanation, no markdown fences.`;

  const response = await client.chat.completions.create({
    model: "gpt-4o",
    messages: [
      { role: "system", content: systemPrompt },
      { role: "user", content: prompt },
    ],
    temperature: 0,
  });

  return response.choices[0].message.content?.trim() ?? "";
}
```

## Test File (`dsl-e2e.test.ts`)

```ts
// @vitest-environment jsdom
import { render, screen } from "@testing-library/react";
import React from "react";
import { beforeAll, describe, expect, it, vi } from "vitest";
import { createParser } from "@openuidev/lang-core";
import { Renderer } from "@openuidev/react-lang";
import { dslLibrary } from "../../genui-lib/dslLibrary";
import { fixtures } from "./fixtures";
import { loadOrGenerate } from "./llm";

vi.mock("echarts", () => ({
  init: vi.fn(() => ({
    setOption: vi.fn(),
    dispose: vi.fn(),
    resize: vi.fn(),
  })),
  registerTheme: vi.fn(),
}));

const parser = createParser(dslLibrary.toJSONSchema());
const spec = dslLibrary.toSpec();

describe.each(Object.entries(fixtures))("%s", (component, scenarios) => {
  it.each(scenarios)("$id", async ({ id, prompt, dataModel, assert }) => {
    const dsl = await loadOrGenerate(id, prompt, dataModel, spec);

    // Parse layer: LLM output must be valid DSL
    const parsed = parser.parse(dsl);
    expect(parsed.meta.errors, `parse errors in ${id}: ${dsl}`).toHaveLength(0);

    // Render layer: component must not crash and contain expected markers
    const { container } = render(
      <Renderer library={dslLibrary} response={dsl} dataModel={dataModel} />,
    );
    for (const text of assert.contains) {
      expect(container.innerHTML).toContain(text);
    }
  });
});
```

## Error Handling

- If `OPENAI_API_KEY` is not set and a snapshot is missing, the test fails with a clear message: `"Snapshot missing and OPENAI_API_KEY is not set. Run test:e2e:regen with a valid key."`
- Parse errors include the full DSL string in the failure message to aid debugging
- LLM responses that return empty strings fail immediately with a descriptive error

---

## What This Does Not Cover

- Visual/CSS regression (no pixel comparison)
- Interactive behavior (clicks, form submission)
- ECharts canvas output (mocked)
- Components outside Table and Charts (Button, Form, Card, etc.)
