## Context

The openui-lang expression grammar is a hand-written Pratt parser (`expressions.ts`) with a flat-array lexer (`lexer.ts`). The `?` character currently emits `T.Question` and is only used for ternary (`cond ? then : else`). When the LLM writes `??`, the lexer emits two consecutive `T.Question` tokens; the parser interprets the first as a ternary opener, the second falls into the unknown-prefix path and returns `{k:"Null"}`, silently producing `cond ? null : right` — wrong behavior with no error signal.

Benchmark run `20260428_102409_kd0s` (47 fixtures) confirmed: `?.` optional chaining appears 0 times in LLM output, so scope is `??` only. The runtime `Member` evaluator already returns `null` when `obj == null`, so plain `.` is already null-safe and `?.` adds no value.

## Goals / Non-Goals

**Goals:**
- Add `??` as a first-class infix operator with correct short-circuit semantics
- Fix the silent mis-parse of `??` into broken ternary
- Enable clean null-fallback expressions: `v ?? "—"`, `(row.cpuCores ?? 0) + " 核"`

**Non-Goals:**
- `?.` optional chaining — not generated by LLM, runtime already null-safe
- Changing the precedence of any existing operator
- Any change outside `packages/lang-core`

## Decisions

### D1: New `T.NullCoal` token vs. parser-level double-Question detection

**Decision:** Add `T.NullCoal = 36` to the token enum and handle it in the lexer.

**Rationale:** The lexer is the right place to distinguish `??` from `?`. Doing it in the parser requires peeking at two consecutive `T.Question` tokens and consuming them as a unit — complicating the parser for a concern that belongs at tokenization. A dedicated token keeps each layer's responsibility clean.

**Alternative rejected:** Parser-level peek — adds stateful lookahead to an otherwise single-token-lookahead Pratt parser.

### D2: Precedence `PREC_NULLCOAL = PREC_OR = 2`

**Decision:** Give `??` the same precedence level as `||` (both = 2), left-associative.

**Rationale:** No existing constant needs renumbering. In practice, LLM-generated `??` operands are always parenthesized (`(v ?? "—") + " 核"`), so the exact precedence relative to `||` has no observable effect on real DSL output. Matching JavaScript's strict rule (no mixing `??` with `||`/`&&` without parens) is not worth the churn of inserting a new precedence level.

**Alternative rejected:** Insert `PREC_NULLCOAL = 2`, shift OR to 3, AND to 4, etc. — 5 constant changes, zero practical benefit.

### D3: Short-circuit evaluation — null and undefined only

**Decision:** Right side is only evaluated when left is `null` or `undefined`. Falsy values (`0`, `""`, `false`) return the left side unchanged.

**Rationale:** This is the semantic difference between `??` and `||`. In dashboard DSL contexts, `0` (e.g. a metric value of zero) and `""` (empty string fields) are valid data values that should not trigger fallback. Using `||` semantics would silently discard legitimate zeros.

## Risks / Trade-offs

- **Precedence same as `||`**: `a || b ?? c` is technically ambiguous (same level, left-assoc, so parses as `(a || b) ?? c`). In JavaScript this is a syntax error. In our DSL it silently works but may surprise — acceptable given LLM-generated DSL never mixes them without parens.
- **`T.NullCoal = 36`**: The `T` enum uses a `const enum` (inlined at compile time). Adding `36` at the end has zero runtime cost and no backward-compat concern since the enum is internal.

## Open Questions

None — all decisions resolved in brainstorming.
