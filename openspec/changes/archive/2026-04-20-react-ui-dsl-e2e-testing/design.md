## Context

`packages/react-ui-dsl` exports 15 DSL components built on Ant Design and ECharts. They are consumed by passing an `openui-lang` DSL string to `<Renderer library={dslLibrary} response={dsl} dataModel={...} />`. In production, the DSL string is generated by an LLM given a system prompt derived from `dslLibrary.toSpec()`.

Two failure modes currently go undetected:

1. **Parse errors** — the LLM generates DSL that fails `createParser` schema validation (wrong argument order, missing required props, unrecognised component names).
2. **Render errors** — the component tree crashes or produces incorrect output when the parsed result is handed to React.

The only existing test (`Table/index.test.tsx`) tests `mapColumnsToAntd` in isolation. `react-lang` has Renderer tests using `renderToStaticMarkup`, but those tests use a toy two-component library, not `dslLibrary`.

The most error-prone components are the data-driven ones: **Table** (complex columnar schema with options) and **Charts** (PieChart, LineChart, BarChart, GaugeChart, all backed by ECharts via `useEffect`).

## Goals / Non-Goals

**Goals:**

- Cover the full `LLM output → parse → render` pipeline for Table and all four Chart components.
- Use real LLM-generated DSL as test input so tests catch the same failure modes that appear in production.
- Store snapshots as committed plain-text `.dsl` files so tests are deterministic in CI and diffs are human-readable.
- Allow snapshot regeneration on demand (`REGEN_SNAPSHOTS=1`) without changing any test assertions.
- Follow community-standard vitest patterns (`describe.each`, `it.each`, per-file `jsdom` override).

**Non-Goals:**

- Visual or pixel-level regression testing.
- Testing interactive behaviour (clicks, form submission, chart hover).
- Covering ECharts canvas rendering (mocked at the `echarts` module level).
- Expanding coverage to Button, Form, Card, List, or layout components in this change.

## Decisions

### VCR snapshot model over hand-crafted DSL

Tests are driven by LLM-generated `.dsl` files that are committed to git. On a clean checkout, all snapshots exist and no LLM is called. `REGEN_SNAPSHOTS=1` forces regeneration.

Alternative considered: hand-craft the DSL strings inline in each test.
Rejected because hand-crafted DSL does not catch the LLM-specific syntax mistakes that cause real production failures. The value of the test comes from the input being generated by the same model that generates it in production.

### `@testing-library/react` + jsdom over `renderToStaticMarkup`

Each test file opts into jsdom with `// @vitest-environment jsdom` so `useEffect` hooks run, the DOM is real, and `render()` / `screen` assertions work as expected.

Alternative considered: `renderToStaticMarkup` (used in `react-lang` tests).
Rejected because `renderToStaticMarkup` does not execute `useEffect`. For Chart components, this means `echarts.init` never runs and the component is effectively not tested. The user explicitly called this out as insufficient.

### Mock `echarts` and `ResizeObserver` rather than polyfill

`vi.mock('echarts')` stubs `init`, `registerTheme`. `ResizeObserver` is not available in jsdom and is polyfilled with a no-op in the test setup file. This lets Chart render tests verify that the container div mounts and props reach the component without requiring a canvas environment.

Alternative considered: use a canvas polyfill (e.g. `jest-canvas-mock`).
Rejected because canvas polyfills are brittle and the goal is not to test ECharts internals.

### `dataModel` uses realistic API response shapes

Each fixture's `dataModel` matches a plausible backend API response (nested objects, arrays, ISO timestamps). The LLM is prompted to map `data.xxx` paths to component props. This tests the LLM's ability to handle real data shapes, not a purpose-built payload.

Alternative considered: craft `dataModel` to exactly match what each component expects.
Rejected because that removes the realism that makes the tests valuable as e2e tests.

### OpenAI SDK for LLM calls

`openai` is used with `new OpenAI({ apiKey: process.env.OPENAI_API_KEY })`, consistent with `examples/*/api/chat/route.ts` throughout the monorepo.

## Architecture

```
packages/react-ui-dsl/src/__tests__/e2e/
├── setup.ts              # ResizeObserver no-op polyfill; wired via vitest.config.ts setupFiles
├── fixtures.ts           # Fixture[] grouped by component name
├── snapshots/            # Committed .dsl files, one per fixture id
│   ├── table-basic.dsl
│   ├── table-sortable-date.dsl
│   ├── pie-sales-by-region.dsl
│   ├── line-monthly-revenue.dsl
│   ├── bar-product-comparison.dsl
│   └── gauge-kpi.dsl
├── llm.ts                # loadOrGenerate() + callLLM() using openai SDK
└── dsl-e2e.test.ts       # describe.each / it.each test file
```

**Test execution flow:**

```
loadOrGenerate(id, prompt, dataModel, spec)
  ├─ snapshot exists and REGEN_SNAPSHOTS unset → readFileSync → return dsl
  └─ otherwise → callLLM(prompt, dataModel, spec) → writeFileSync → return dsl

it.each(scenarios)
  ├─ parser.parse(dsl)         → expect(errors).toHaveLength(0)
  └─ render(<Renderer .../>)   → expect(container.innerHTML).toContain(...)
```

**LLM system prompt construction:**

```
Available components:
<dslLibrary.toSpec() as JSON>

Runtime data available as data.xxx:
<JSON.stringify(dataModel)>

Rules:
- Reference data using data.fieldName paths
- Return ONLY the DSL, no explanation, no markdown fences
```

## Risks / Trade-offs

- [jsdom does not have `ResizeObserver`] → polyfill with a no-op in `setup.ts`; add `setupFiles` entry to `vitest.config.ts`.
- [antd may emit console warnings in jsdom] → suppress with `vi.spyOn(console, 'error')` in setup if they become noise; do not suppress unconditionally.
- [LLM output quality varies by model] → use `temperature: 0` and a precise system prompt; if a snapshot fails parse, the test output includes the full DSL string for debugging.
- [Snapshot files can go stale after component schema changes] → `test:e2e:regen` is the documented recovery path; schema changes should trigger a regen as part of the PR checklist.
- [OpenAI API key not available in CI] → CI only runs `test:e2e` against committed snapshots; `OPENAI_API_KEY` is never required for the standard test run.

## Migration Plan

1. Install `@testing-library/react` and `openai` as dev dependencies in `packages/react-ui-dsl`.
2. Create `src/__tests__/e2e/setup.ts` with `ResizeObserver` polyfill; wire it into `vitest.config.ts` via `setupFiles`.
3. Create `fixtures.ts` with the initial scenario set (2 Table + 1 per chart type = 6 fixtures).
4. Implement `llm.ts` with `loadOrGenerate` and `callLLM`.
5. Implement `dsl-e2e.test.ts`.
6. Run `test:e2e:regen` once to generate all `.dsl` snapshots; commit them.
7. Verify `test:e2e` passes on the committed snapshots without `OPENAI_API_KEY`.
8. Add `test:e2e` to CI.

## Open Questions

- Should the system prompt sent to the LLM use `dslLibrary.toSpec()` (structured JSON) or `dslLibrary.toPrompt()` if a human-readable prompt format becomes available?
- Should `test:e2e:regen` regenerate all snapshots unconditionally, or only those whose fixture definition has changed (e.g. by hashing fixture metadata)?
