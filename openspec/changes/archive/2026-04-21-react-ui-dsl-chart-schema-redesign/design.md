## Context

`react-ui-dsl` is the DSL layer for a network management system UI. Charts are generated by an LLM from API data. The current chart schemas pass raw ECharts config as props, requiring the LLM to know ECharts internals. The `react-ui` package (recharts-based) already has a well-designed genui-lib with domain-friendly schemas (`labels/series`) — this change ports that pattern to `react-ui-dsl` (ECharts-based).

The stack is: LLM generates DSL text → DSL parser evaluates props against Zod schemas → React component renders via ECharts.

## Goals / Non-Goals

**Goals:**
- Replace all chart schemas with semantic, typed fields — no `z.any()`
- Match the `labels/series` pattern from `react-ui/src/genui-lib/Charts/` for consistency
- Add 6 new chart types relevant to network management (HorizontalBarChart, AreaChart, RadarChart, HeatmapChart, TreeMapChart, ScatterChart)
- Keep `components/chart/` (BaseChart, theme) completely unchanged

**Non-Goals:**
- Switching rendering engine (ECharts stays)
- Adding Table `cellType` mini-chart support (deferred)
- Adding FunnelChart, Sankey, Waterfall, or Candlestick

## Decisions

### D1: genui-lib layer only — leave components/chart untouched

The ECharts conversion happens in each chart's `view/index.tsx`, not in the base component. This isolates the breaking schema change from the rendering layer and keeps `BaseChart` stable.

Alternatives considered: Refactor `components/chart/BarChart` etc. to accept `labels/series` directly — rejected because it expands blast radius with no LLM-generation benefit (the base components aren't schema-visible to the LLM).

### D2: Mirror react-ui virtual component pattern (Series, ScatterSeries, Point)

Virtual components are `defineComponent` entries with `component: () => null`. They appear in the DSL schema/signature so the LLM knows their shape, but render nothing. The same pattern exists in `react-ui/src/genui-lib/Charts/Series.ts`.

`series.category` (not `series.name`) matches react-ui's field name for consistency across packages.

### D3: ScatterChart uses `datasets: ScatterSeriesSchema[]` (mirror react-ui)

react-ui uses `datasets` (not `series`) for scatter to distinguish from the `SeriesSchema` type used by multi-series charts. Keeping the same field name reduces cognitive switching cost.

### D4: HeatmapChart values are `number[][]` indexed as `values[yIndex][xIndex]`

Row-major order with y as outer dimension matches ECharts' heatmap data format after flattening. The view layer flattens to `[x, y, value]` tuples as required by ECharts.

### D5: TreeMapChart uses flat array with optional `group` field

`{ name, value, group? }[]` is simpler for LLM generation than a recursive tree structure. The view layer groups by `group` to build ECharts `children` nesting. For network management, one level of grouping (e.g., subnet → devices) covers the primary use case.

### D6: LineChart/AreaChart variant is `"linear" | "smooth" | "step"`

react-ui uses `"linear" | "natural" | "step"` (recharts terminology). ECharts uses `smooth: true` for natural curves. Renaming to `"smooth"` avoids exposing recharts vocabulary in an ECharts component.

## Risks / Trade-offs

**BREAKING schema change** → Existing DSL snapshots in `__tests__/e2e/snapshots/` will fail until regenerated. Mitigation: regenerate all chart snapshots as part of this change; update E2E fixtures to use new schema.

**ECharts heatmap requires `visualMap` component** → Auto-generated visualMap range from `min/max` of values; no user-facing prop needed. Risk of poor color scale on skewed data. Mitigation: use `calculable: true` on visualMap to allow user drag-adjustment at runtime.

**TreeMap single-level grouping** → Deep hierarchies (subnet → rack → device) can't be fully represented. Accepted trade-off for LLM simplicity; power users can still use the raw ECharts component directly.

## Migration Plan

1. Update schemas (breaking) — old `data: { source: number[][] }` props no longer accepted
2. Regenerate E2E snapshots by running `vitest` after implementation
3. No external API changes — `components/chart/` public interface unchanged
